Introduction

A REST API for scheduling and publishing posts to X (Twitter) across teams and accounts.

The Tempo API lets you connect X accounts, compose and queue posts, define recurring schedules, and publish automatically. It powers the Tempo dashboard and is the same surface a script, agent, or integration would call.

Base URL

# replace with your deployment origin
https://tempoapp.hq/api

Channels

One connected X account. Every request is scoped to a channel and gated by your membership role.

Queue & Posts

Posts live in a per-channel queue, auto-slotted into the next opening or pinned to a time.

Schedule & Automations

A weekly schedule defines when the queue drains; rules materialize recurring posts.

Publishing

A cron engine publishes due posts, refreshes OAuth tokens, retries, and audit-logs everything.

Quickstart

List your channels and queue your first post in three requests.

1 · Set your token

export TEMPO_TOKEN="<supabase-access-token>"
export TEMPO_BASE="https://tempoapp.hq/api"

2 · List channels

curl "$TEMPO_BASE/channels" \
  -H "Authorization: Bearer $TEMPO_TOKEN"

3 · Queue a post

curl -X POST "$TEMPO_BASE/posts" \
  -H "Authorization: Bearer $TEMPO_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "channel_id": "8f3c...", "text": "Hello from the Tempo API" }'

TypeScript SDK

Generated by Fern from the same spec, so it never drifts.

import { TempoClient } from "@tempo/sdk";

const tempo = new TempoClient({ token: process.env.TEMPO_TOKEN });

const channels = await tempo.channels.list();
await tempo.posts.create({
  channelId: channels[0].id,
  text: "Hello from the Tempo API",
});
Omit scheduled_at and Tempo auto-assigns the next open slot. Pin a time and a taken slot returns 409.

Authentication

Authenticate user endpoints with a Supabase-issued Bearer JWT. The server validates the token, then confirms you are a member of the target channel.

Authorization: Bearer <supabase-access-token>
RoleCan do
memberRead channel data; create and manage posts
admin+ manage categories and the schedule
owner+ manage team members and alerts
Internal /cron/* endpoints authenticate with a shared CRON_SECRET instead of a user token.

Scheduling & Slots

Posts occupy discrete time slots from the channel's weekly schedule. A database-level uniqueness guarantee allows one active post per slot, so pinning to a taken slot returns:

{ "error": "That time slot is already taken." }   // 409 Conflict

Errors & Status Codes

Every endpoint returns one predictable shape: { "error": "..." }.

CodeMeaningTypical cause
400Bad requestMissing required field
401UnauthorizedMissing/invalid token
403ForbiddenNot a member, or role too low
404Not foundResource does not exist
409ConflictScheduling slot already taken
